def huffman_tree():
    import heapq
    frequencies = {'A': 40, 'P': 9, 'Q': 2, 'L': 10, 'E': 16, 'S': 13}
    heap = [[weight, [char, ""]] for char, weight in frequencies.items()]
    heapq.heapify(heap)
    print("Initial heap:", heap)
    
    # Let's do it manually or step-by-step to see the structure
    # Nodes can be represented as [weight, left_child, right_child] or similar.
    # To keep track of structure, let's build a proper tree.
    class Node:
        def __init__(self, weight, char=None, left=None, right=None):
            self.weight = weight
            self.char = char
            self.left = left
            self.right = right
        def __lt__(self, other):
            return self.weight < other.weight

    nodes = [Node(weight, char=char) for char, weight in frequencies.items()]
    heapq.heapify(nodes)
    
    step = 1
    while len(nodes) > 1:
        left = heapq.heappop(nodes)
        right = heapq.heappop(nodes)
        parent = Node(left.weight + right.weight, left=left, right=right)
        heapq.heappush(nodes, parent)
        print(f"Step {step}: Combined {left.char if left.char else f'internal({left.weight})'} and {right.char if right.char else f'internal({right.weight})'} -> internal({parent.weight})")
        step += 1
        
    root = nodes[0]
    
    # Traverse tree to find codes
    codes = {}
    def traverse(node, code):
        if node.char:
            codes[node.char] = code
            return
        traverse(node.left, code + "0")
        traverse(node.right, code + "1")
    
    traverse(root, "")
    print("Codes:", codes)

huffman_tree()

# ## Part (a): Text Compression using Huffman Coding

### 1. Selection of a Suitable Coding System

The most suitable coding system for this scenario is **Huffman Coding**.

**Justification:**

- **Lossless Compression:** It ensures that the original text can be perfectly reconstructed without any data loss.
    
- **Variable-Length Prefix Coding:** It assigns shorter binary codes to more frequent characters and longer codes to less frequent characters, minimizing the total number of bits required.
    
- **Prefix Property:** No code is a prefix of another code, which prevents ambiguity during the decoding phase.
    

### 2. Step-by-Step Construction of the Huffman Tree

We start by listing the characters sorted in ascending order of their frequencies:

$$\text{Q: 2, \, P: 9, \, L: 10, \, S: 13, \, E: 16, \, A: 40}$$

- **Step 1:** Combine the two lowest frequency nodes, **Q (2)** and **P (9)**, into an internal node $N_1$ with weight $2 + 9 = 11$.
    
    - _Remaining active nodes:_ L (10), $N_1$ (11), S (13), E (16), A (40)
        
- **Step 2:** Combine the next two lowest nodes, **L (10)** and **$N_1$ (11)**, into an internal node $N_2$ with weight $10 + 11 = 21$.
    
    - _Remaining active nodes:_ S (13), E (16), $N_2$ (21), A (40)
        
- **Step 3:** Combine the next two lowest nodes, **S (13)** and **E (16)**, into an internal node $N_3$ with weight $13 + 16 = 29$.
    
    - _Remaining active nodes:_ $N_2$ (21), $N_3$ (29), A (40)
        
- **Step 4:** Combine the next two lowest nodes, **$N_2$ (21)** and **$N_3$ (29)**, into an internal node $N_4$ with weight $21 + 29 = 50$.
    
    - _Remaining active nodes:_ A (40), $N_4$ (50)
        
- **Step 5:** Combine the final two remaining nodes, **A (40)** and **$N_4$ (50)**, into the root node with total weight $40 + 50 = 90$.
    

#### **The Resulting Huffman Tree Structure:**

_(By convention, Left branches are assigned `0` and Right branches are assigned `1`)_

```
             (90)
            /    \
        0  /      \  1
        A(40)    (50)
                /    \
            0  /      \  1
            (21)      (29)
           /    \     /    \
       0  /    1 \   / 0    \ 1
       L(10)   (11) S(13)  E(16)
              /    \
          0  /      \ 1
           Q(2)    P(9)
```

### 4. Encoding the Message "APPLE"

To encode the message, we look up each letter's corresponding Huffman code:

$$\text{A} \rightarrow 0$$

$$\text{P} \rightarrow 1011$$

$$\text{P} \rightarrow 1011$$

$$\text{L} \rightarrow 100$$

$$\text{E} \rightarrow 111$$

- **Encoded Bitstream:** `010111011100111`
    

### 5. Decoding the Bitstream Back to Original Message

The decoder reads the bitstream `010111011100111` from left to right, traversing the tree starting from the root node for each character:

1. **`0`** $\rightarrow$ Moves Left $\rightarrow$ Reaches leaf **A**. _(Output: **A**)_
    
2. **`1`** $\rightarrow$ Right, **`0`** $\rightarrow$ Left, **`1`** $\rightarrow$ Right, **`1`** $\rightarrow$ Right $\rightarrow$ Reaches leaf **P**. _(Output: **P**)_
    
3. **`1`** $\rightarrow$ Right, **`0`** $\rightarrow$ Left, **`1`** $\rightarrow$ Right, **`1`** $\rightarrow$ Right $\rightarrow$ Reaches leaf **P**. _(Output: **P**)_
    
4. **`1`** $\rightarrow$ Right, **`0`** $\rightarrow$ Left, **`0`** $\rightarrow$ Left $\rightarrow$ Reaches leaf **L**. _(Output: **L**)_
    
5. **`1`** $\rightarrow$ Right, **`1`** $\rightarrow$ Right, **`1`** $\rightarrow$ Right $\rightarrow$ Reaches leaf **E**. _(Output: **E**)_
    

- **Decoded Message:** `APPLE`
    

# ## Part (b): Divide & Conquer and Binary Search

### 1. Control Abstraction for Divide and Conquer Method

The control abstraction characterizes the generic execution flow of the **Divide and Conquer** paradigm:


```  
Algorithm DandC(P)
{
    if Small(P) then
        return Solution(P); // Solves directly for base case
    else
    {
        // Divide the problem into 'k' smaller independent subproblems
        divide P into smaller subproblems P1, P2, ..., Pk;
        
        // Conquer by recursively solving each subproblem
        for i := 1 to k do
            Si := DandC(Pi);
            
        // Combine the sub-solutions into the final solution
        return Combine(S1, S2, ..., Sk);
    }
}
```

### 2. Binary Search under Divide and Conquer Principle

Binary Search utilizes a specialized optimization of Divide and Conquer known as **Decrease and Conquer** (specifically, prune-and-search):

1. **Divide:** The search range is split into two halves by locating the middle index: $\text{mid} = \lfloor(\text{low} + \text{high})/2\rfloor$.
    
2. **Conquer:** The target element is compared with the element at $A[\text{mid}]$. If it matches, the algorithm terminates. If the target is smaller, the search recursively proceeds _only_ in the left half. If greater, it proceeds _only_ in the right half.
    
3. **Combine:** Unlike classic algorithms (such as Merge Sort), there is **no combination phase** necessary because identifying the element in a sub-segment directly serves as the complete solution.
    

### 3. Recursive Algorithm for Binary Search

Delphi

```
Algorithm BinarySearch(A, low, high, target)
// Input: Sorted array A, boundary bounds low and high, element to find
// Output: Index of the target if found, otherwise -1
{
    if (low > high) then
        return -1; // Base Case: Target element is not present
        
    mid := floor((low + high) / 2);
    
    if (A[mid] = target) then
        return mid; // Base Case: Element found
    else if (target < A[mid]) then
        return BinarySearch(A, low, mid - 1, target); // Recursive Case: Left sub-array
    else
        return BinarySearch(A, mid + 1, high, target); // Recursive Case: Right sub-array
}
```

### 4. Recurrence Relation and Time Complexity Analysis

#### **Recurrence Relation Setup:**

In each execution cycle, the problem size $n$ reduces by half with a constant processing overhead $c$ for mid-point computation and condition checks:

$$T(n) = T\left(\frac{n}{2}\right) + c \quad \text{for } n > 1$$

$$T(1) = 1$$

#### **Solving via Substitution Method:**

Substitute the relation iteratively:

$$T(n) = T\left(\frac{n}{2}\right) + c$$

$$T(n) = \left[T\left(\frac{n}{4}\right) + c\right] + c = T\left(\frac{n}{4}\right) + 2c$$

$$T(n) = T\left(\frac{n}{8}\right) + 3c$$

Following this pattern for $k$ structural expansion steps:

$$T(n) = T\left(\frac{n}{2^k}\right) + k \cdot c$$

The base condition is reached when the array size reduces to 1:

$$\frac{n}{2^k} = 1 \implies n = 2^k \implies k = \log_2 n$$

Substitute $k = \log_2 n$ back into our expanded equation:

$$T(n) = T(1) + c \cdot \log_2 n$$

$$T(n) = 1 + c \cdot \log_2 n$$

$$\therefore \text{Worst-case Time Complexity} = O(\log_2 n)$$
