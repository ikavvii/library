### **Question 3 (25 Marks Total)**

#### **Extracted Questions**

- **Subdivision A (3 Marks):** An e-learning platform wants to show live quiz results updating in real time. How would **Virtual DOM** help ensure smooth rendering without reloading the entire page?
- **Subdivision B (10 Marks):** Evaluate the suitability of **Single Page Applications (SPA) and Multi-Page Applications (MPA)** for the application (**ShopSphere**, an online marketplace). Discuss in terms of performance, scalability, user experience, and security. Which approach would you recommend? Justify your answer.
- **Subdivision C (12 Marks):** Analyze the use of **MEAN and MERN stacks** for the development of **BookHive**, a social networking site for book lovers. Compare their architecture, strengths, and challenges in terms of performance, scalability, developer productivity, and real-time interaction support. Recommend which stack is more suitable and justify your answer.

#### **Detailed Solutions**

**A. Virtual DOM for Real-Time Quiz Results [3 Marks]** The **Virtual DOM (VDOM)** is a lightweight, in-memory representation of the actual DOM. For an e-learning platform's live quiz:

- **Efficient Updates:** When results change, the VDOM uses a **diffing algorithm** to compare the new state with the previous one, identifying exactly which score or chart needs updating.
- **Optimized Rendering:** Instead of reloading the entire page or re-rendering the whole result table, it updates only the specific nodes in the real DOM. This ensures a **smooth, lag-free experience** for students tracking live scores.
- **Batching:** Multiple updates (e.g., several students finishing at once) are batched into a single real DOM update, reducing browser overhead.

**B. SPA vs. MPA for "ShopSphere" Marketplace [10 Marks]**

- **Performance:** **SPAs** have slower initial loads but extremely fast subsequent interactions because only data is exchanged. **MPAs** load quickly initially but every action (like switching product categories) requires a full server round-trip.
- **Scalability:** MPAs scale well for large catalogs with thousands of products because each page has its own URL, making them **SEO-friendly**. SPAs are more scalable in terms of handling complex, dynamic user interactions like a persistent shopping cart.
- **User Experience (UX):** SPAs offer a fluid, **app-like experience** ideal for browsing products and checking out seamlessly. MPAs can feel "choppy" due to constant page refreshes.
- **Security:** MPAs are generally easier to secure for **payment processing** using standard server-side redirects. SPAs require advanced client-side security to protect sensitive data.
- **Recommendation:** For a modern marketplace like ShopSphere, an **SPA (or a hybrid approach)** is recommended to provide the high responsiveness required for browsing and the cart experience, provided robust client-side security and server-side rendering (for SEO) are implemented.

**C. MEAN vs. MERN Stack for "BookHive" [12 Marks]**

- **Architecture:** **MEAN** uses Angular (a comprehensive framework), while **MERN** uses React (a flexible library focused on the View layer).
- **Real-Time Interaction:** **MERN** is often superior for "BookHive" because its Virtual DOM handles the high frequency of **real-time notifications** for new comments and likes more efficiently.
- **Productivity:** MERN typically offers higher developer productivity for social features due to its vast ecosystem of reusable components. MEAN provides a more structured environment which is beneficial for large teams maintaining strict standards.
- **Scalability:** Both stacks are highly scalable due to the **Node.js** backend and **MongoDB**.
- **Recommendation:** **MERN** is recommended for BookHive because its component-based architecture and efficient VDOM updates are better suited for the highly interactive, "feed-based" nature of a social networking site.

### **Question 4 (25 Marks Total)**

#### **Extracted Questions**

- **Subdivision A (3 Marks):** List any two key characteristics of **Microservices architecture**.
- **Subdivision B (10 Marks):** Differentiate **synchronous and asynchronous communication** in distributed systems. Discuss how **middleware** supports communication and interoperability.
- **Subdivision C (12 Marks - Option i):** 1) Illustrate how **SOAP, WSDL, and UDDI** work together in a web services stack. 2) For an **insurance management system** integrating with external policy verification, describe how to design, develop, deploy, and consume **Java SOAP Web Services**.
- **Subdivision C (12 Marks - Option ii):** 1) Describe the use of **HTTP methods** as actions in RESTful services. 2) Suggest a **RESTful web service design** for a weather forecasting agency providing real-time updates and alerts.

#### **Detailed Solutions**

**A. Characteristics of Microservices [3 Marks]**

1. **Independent Deployability:** Each service can be deployed, updated, and scaled independently without affecting the rest of the system.
2. **Single Responsibility:** Each microservice focuses on a single, well-defined business function (e.g., "Payment Service" or "Inventory Service").

**B. Sync vs. Async Communication and Middleware [10 Marks]**

- **Synchronous Communication:** The client sends a request and **waits for a response** before continuing (e.g., a standard HTTP REST call). It is simple but can lead to bottlenecks if the server is slow.
- **Asynchronous Communication:** The client sends a message and **does not wait**, continuing its task immediately (e.g., using message queues). This decouples the systems, improving resilience and performance.
- **Middleware Role:** Middleware acts as a "glue" layer that facilitates communication between different platforms. It handles data translation, security, and message routing, ensuring **interoperability** between a Java-based backend and a third-party service.

**C. Option (i): SOAP Stack and Insurance Integration [12 Marks]**

- **SOAP Stack Interaction:**
    - **SOAP:** The messaging protocol for data exchange.
    - **WSDL:** The XML document describing the service's methods and data types (the "contract").
    - **UDDI:** The directory where services are registered and discovered.
- **Insurance System Case:**
    - **Design:** Define a secure XML structure for policy data using **JAX-WS**.
    - **Develop/Deploy:** Implement the service in Java, generate the WSDL, and deploy it to a secure application server like GlassFish.
    - **Consume:** The third-party verification service uses the WSDL to generate a client stub and securely invokes the remote methods to verify policies.

**C. Option (ii): REST Actions and Weather API Design [12 Marks]**

- **HTTP Methods as Actions:**
    - **GET:** Retrieve weather data (e.g., current temp).
    - **POST:** Submit new weather observations.
    - **PUT:** Update existing historical data.
    - **DELETE:** Remove outdated alerts.
- **Weather Agency API Design:**
    - **Resources:** Use intuitive URIs like `/weather/current/{city}`, `/weather/history/{date}`, and `/weather/alerts`.
    - **Justification:** Using **JSON** over REST ensures the data is lightweight and easily consumable by both web dashboards and mobile apps, providing the low latency required for "real-time" alerts.
