### **Question 3: Streams and Multithreading (CO3)**

#### **a) i) Compare extending the Thread class and implementing the Runnable interface. Is it better to extend Thread or implement Runnable? (3 Marks)**

- **Comparison:**
    - **Inheritance:** When you **extend the `Thread` class**, your class cannot extend any other class because Java does not support multiple inheritance. **Implementing `Runnable`** allows your class to extend another base class while still being a thread.
    - **Coupling:** Extending `Thread` creates a **tight coupling** between your task and the thread management logic. Implementing `Runnable` promotes **loose coupling** as the task is separated from the execution mechanism.
    - **Resource Sharing:** `Runnable` is better for sharing the same object instance among multiple threads.
- **Recommendation:** It is generally **better to implement the `Runnable` interface**. This approach provides more flexibility for class hierarchies and is essential when using modern thread management tools like thread pools.

#### **b) i) Write a program using FileReader and FileWriter to copy contents of the file "test.in" to the file "answer.out". (5 Marks)**

```
import java.io.*;

public class FileCopy {
    public static void main(String[] args) {
        // Use try-with-resources for automatic closing of streams
        try (FileReader fr = new FileReader("test.in");
             FileWriter fw = new FileWriter("answer.out")) {

            int character;
            // read() returns -1 at the end of the file
            while ((character = fr.read()) != -1) {
                fw.write(character);
            }
            System.out.println("File copied successfully.");
        } catch (FileNotFoundException e) {
            System.err.println("Source file not found: " + e.getMessage());
        } catch (IOException e) {
            System.err.println("I/O error occurred: " + e.getMessage());
        }
    }
}
```

#### **c) i) Distinguish the Reader/Writer class hierarchy and InputStream/OutputStream class hierarchy. Can we say the file streams in Java work on the principles of object serialization? Justify. (12 Marks)**

- **Hierarchy Comparison:**
    - **InputStream/OutputStream (Byte Streams):** These are abstract base classes for handling **8-bit binary data**. They are used for processing raw data like images, audio, video, and executable files.
    - **Reader/Writer (Character Streams):** These are abstract base classes for handling **16-bit Unicode characters**. They were introduced later to provide robust internationalization support for text files.
- **Serialization Principles:**
    - **No**, basic file streams (like `FileInputStream`) do **not** work on the principles of object serialization.
    - **Justification:** Basic file streams handle data at the level of raw bytes or characters. **Object Serialization** is a specific process using `ObjectOutputStream` and `ObjectInputStream` to convert a whole **object state** into a byte stream (and vice versa). While serialization _uses_ an underlying stream, the basic file streams themselves are just "pipes" for raw data and do not understand object structures or the `Serializable` interface.

#### **c) ii) Illustrate thread's life cycle with a diagram and relevant method calls. (12 Marks)**

The Java thread lifecycle consists of five primary states:

1. **New (Born):** The thread is created using `new Thread()` but `start()` is not yet called.
2. **Runnable (Ready):** The `start()` method is invoked. The thread is ready to run and waiting for the CPU scheduler to pick it.
3. **Running:** The CPU scheduler has assigned time to the thread, and it is executing its `run()` method.
4. **Blocked / Waiting (Non-Runnable):** The thread is alive but ineligible to run. This happens due to method calls like `Thread.sleep(ms)`, `wait()`, or waiting for I/O. It moves back to **Runnable** when the sleep time expires or `notify()` is called.
5. **Terminated (Dead):** The `run()` method finishes or the `stop()` method (deprecated) is called. A dead thread cannot be restarted.

### **Question 4: User Interfacing, Collections, and Database (CO4)**

#### **a) State the need for garbage collection. Which part of the memory is involved in Garbage Collection? (3 Marks)**

- **Need:** Garbage collection is essential to **automatically reclaim memory** occupied by objects that are no longer reachable by the program. This prevents memory leaks and manual memory management errors (like dangling pointers).
- **Memory Part:** It primarily involves the **Heap memory**, which is where Java objects are stored during runtime.

#### **b) i) Discuss Set, Vector, and Hash map in Java Utility Class with relevant examples. (5 Marks)**

- **Set (e.g., `HashSet`):** A collection that contains **no duplicate elements**. Example: Storing unique student IDs.
- **Vector:** A legacy class that implements a **synchronized, thread-safe dynamic array**. Example: A shared list of tasks accessed by multiple threads.
- **HashMap:** A collection that stores data in **key-value pairs**. Example: Mapping employee names (Key) to their salaries (Value).

#### **c) Paraphrase various Event classes and listener interfaces for handling events generated in an Applet. (12 Marks)**

In the **Delegation Event Model**, event sources generate objects representing state changes, which are handled by registered listeners.

- **Event Classes:**
    - **`ActionEvent`:** Generated by clicking a `Button` or selecting a menu item.
    - **`MouseEvent`:** Generated by mouse actions like clicking, dragging, or moving.
    - **`KeyEvent`:** Generated by keyboard input (press, release, type).
    - **`WindowEvent`:** Generated by window actions like closing or minimizing.
- **Listener Interfaces:**
    - **`ActionListener`:** Contains `actionPerformed()` for handling `ActionEvent`.
    - **`MouseListener`:** Handles clicks, entry/exit, and press/release events.
    - **`WindowListener`:** Handles `windowClosing()`, `windowOpened()`, etc..
- **Implementation:** In an applet, you register the listener with the source using methods like `addActionListener(this)` and then implement the interface methods to define the response.
